7 // Na linguagem C, o mecanismo de modularização de um programa em pacotes

9 // primitivo. Implica a manutenção de dois tipos de ficheiros em paralelo: (a)
10 // os ficheiros com as implementações dos módulos físicos (ficheiros em código

15 // interfaces desses módulos físicos, com a extensão `.h`, e aos quais se dá o

52 // ### Protecção contra inclusões múltiplas
53 //
54 // Os ficheiros de cabeçalho são usados através de um mecanismo também ele muito
55 // primitivo: a directiva de inclusão (`#include`) do pré-processador. Este
56 // mecanismo leva a que seja frequente a ocorrência de múltiplas inclusões do
57 // mesmo ficheiro de cabeçalho. Isso é problemático se levar à definição
58 // múltipla de artefactos que podem ser definidos apenas num local. Para evitar
59 // esse problema, é usual recorrer-se à utilização de directivas de pré-
60 // processamento de formas semipadronizadas. É o que fazemos aqui.
61 //
62 // Todo o código do ficheiro de cabeçalho fica envolto numa instrução
63 // condicional do pré-processador. Ou seja, esse código só é considerado pelo
64 // compilador se a macro `ISLA_EDA_ARRAY_UTILS_H_INCLUDED` não estiver definida.
65 // Se isso acontecer, a macro em causa será definida, de modo a que futuras
66 // inclusões (duplicadas) não levem a considerar o código do ficheiro de
67 // cabeçalho uma segunda vez. O nome escolhido para a macro é convencional e
68 // destina-se e evitar colisões com outras macros relativas a outros ficheiros
69 // de cabeçalho ou com identificadores usados por _clientes_ do ficheiro de
70 // cabeçalho, i.e., por programadores que pretendam usar as ferramentas por ele
71 // declaradas, sem estarem necessariamente envolvidos na sua implementação. A
72 // convenção aqui usada é a seguinte:
73 //
74 // - Seguir a convenção usual de usar apenas maiúsculas como nome de macros,
75 // separando as palavras pelo caracteres sublinhado (_underscore_), ou seja,
76 // `_`.
77 //
78 // - Não alterar o nome usado depois de este ser exposto a uma utilização real.
79 // De outra forma, corre-se o risco (mesmo que menor) de regressões, i.e., de
80 // que código funcional deixe de funcionar depois de uma alteração.
81 //
82 // - Usar como prefixo um identificador da organização que produziu a versão
83 // original do ficheiro de cabeçalho. Neste caso usamos `ISLA_EDA`.
84 //
85 // - De seguida, colocar o nome do ficheiro de cabeçalho, neste caso
86 // `ARRAY_UTILS_H`, incluindo possivelmente um prefixo adicional com o nome da
87 // biblioteca a que pertence.
88 //
89 // - Terminar com `_INCLUDED`.
90 //
91 #ifndef ISLA_EDA_ARRAY_UTILS_H_INCLUDED
