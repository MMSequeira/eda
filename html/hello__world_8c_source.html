hello_world.c
1 // `hello_world.c` &ndash; Estrutura de um programa em C
2 // =====================================================
3 //
4 // Este programa ilustra a estrutura básica de um programa escrito na linguagem
5 // C. O objectivo deste programa é imprimir no ecrã o factorial de um número
6 // inteiro introduzido pelo utilizador do programa.
7 //
8 // Avisos!
9 // -------
10 //
11 // De modo a mantê-lo curto, optámos por omitir deste programa todo o código de
12 // verificação de erros do utilizador. _Todos os programas têm de saber lidar
13 // com erros dos seus utilizadores!_ Não o fazer é um erro _grave_. Este
14 // programa está, assim, _gravemente errado_. Não diga que não o avisámos. :-)
15 //
16 // Ficheiro e nome
17 // ---------------
18 //
19 // Este programa consiste numa única _unidade de compilação_. Ou seja,
20 // corresponde a um único ficheiro destinado a ser _compilado_. Os ficheiros em
21 // linguagem C que se destinam a ser compilados possuem convencionalmente a
22 // extensão `.c`. O nome do ficheiro é, assim, `hello_world.c`.
23 //
24 // Este ficheiro não é suficiente para _construir_ o ficheiro executável, pois
25 // recorre a ferramentas da _biblioteca padrão_ do C. Em particular, recorre a
26 // rotinas de leitura e escrita de dados e recorre à _macro_ `EXIT_SUCCESS`.
27 //
28 // A construção do executável
29 // --------------------------
30 //
31 // Os passos usuais na construção de um ficheiro executável a partir de um
32 // programa em linguagem C são os seguintes:
33 //
34 // 1. Pré-processar cada um dos ficheiros `.c` - O pré-processador é um programa
35 // que lê um ficheiro contendo código C e directivas de pré-processamento e
36 // interpreta estas directivas, produzindo um ficheiro contendo linguagem C
37 // apenas (bem como algumas directivas de pré-processamento indicando a
38 // proveniência das linhas de código). A este ficheiro contendo apenas
39 // linguagem C (ou quase), damos o nome de _unidade de compilação_.
40 // Usualmente as unidades de compilação não têm senão uma existência efémera,
41 // podendo mesmo não chegar a existir enquanto ficheiros no sistema de
42 // ficheiros do computador.
43 //
44 // 2. Compilar cada um dos ficheiros resultantes do pré-processamento, ou seja,
45 // compilar cada uma das unidades de compilação. A compilação é o processo de
46 // tradução da linguagem C para a linguagem máquina do computador em causa. O
47 // resultado da compilação de uma unidade de compilação é um _ficheiro
48 // objecto_. Os ficheiros objecto têm, em Linux, a extensão `.o`. Em Windows
49 // é típico a extensão ser `.obj`. Os ficheiros objecto não são ainda
50 // executáveis, pois tipicamente cada ficheiro objecto contêm código máquina
51 // que recorre a ferramentas externas a esse ficheiro objecto.
52 //
53 // 3. Depois de gerados os ficheiros objecto, estes têm de ser fundidos num
54 // único ficheiro executável. A fusão resolve as dependências entre os vários
55 // ficheiro objecto e entre estes e colecções de ficheiros objecto arquivados
56 // em _bibliotecas_. As bibliotecas, em Linux, têm a extensão `.a` (de
57 // _archive_), quando estáticas, ou `.so`, quando dinâmicas. Em Windows as
58 // extensões correspondentes são `.lib` e `.dll`. O resultado de uma fusão
59 // com sucesso é um ficheiro executável. Em Linux os ficheiros executáveis
60 // podem ser qualquer extensão. Em Windows, a extensão de um ficheiro
61 // executável é `.exe`.
62 
63 // Inclusão de ficheiros de cabeçalho
64 // ----------------------------------
65 //
66 // É usual os ficheiros `.c`, ou seja, os ficheiros que são pré-processados, se
67 // iniciem com a inclusão de um conjunto de _ficheiros de cabeçalho_ padrão.
68 // Cada uma destas inclusões é realizada pelo pré-processador quando encontra a
69 // directiva de pré-processamento `#include`. Esta directiva é obrigatoriamente
70 // seguida do nome do ficheiro de cabeçalho a incluir. Este nome coloca-se entre
71 // parênteses agudos (`<>`) quando os ficheiros a incluir se encontram nas
72 // pastas de ficheiros de cabeçalho padrão e entre aspas (`""`) quando estes se
73 // encontram na mesma pasta do ficheiro `.c` em pré-processamento.
74 //
75 // Os ficheiros de cabeçalho, também conhecidos por _ficheiros de interface_,
76 // contêm sobretudo _declarações_ e, ocasionalmente, também definições,
77 // necessárias para que o código presente no ficheiro que os inclui possa
78 // recorrer a ferramentas _definidas_ noutros locais. A diferença entre
79 // _declaração_ e _definição_ é subtil. Geralmente, uma declaração limita-se a
80 // dar a conhecer ao leitor ou ao compilador a existência de uma dada entidade
81 // (e.g., uma rotina ou uma estrutura), sem a definir. Uma definição é sempre
82 // também uma declaração, mas não se limita a declarar a existência de algo:
83 // define-o de forma completa. As declarações estão, por isso, associadas às
84 // interfaces das ferramentas declaradas e a definições à sua implementação.
85 //
86 // Neste caso particular incluímos dois dos ficheiros de cabeçalho da biblioteca
87 // padrão do C, i.e., da biblioteca que acompanha a linguagem C tal como
88 // definida pela sua norma:
89 //
90 // - `stdio.h` &ndash; Declara os procedimentos `printf()` e `scanf()`, usados
91 // respectivamente para escrever no ecrã (ou melhor, no canal de saída) e
92 // para ler do teclado (ou melhor, do canal de entrada).
93 //
94 // - `stdlib.h` &ndash; Define a _macro_ `EXIT_SUCCESS`, que é usada para
95 // assinalar sucesso na execução de um programa, quando este termina, ou
96 // seja, quando se _sai_ do programa.
97 //
98 #include <stdio.h>
99 #include <stdlib.h>
100 
101 // Definição da função `factorial()`
102 // ---------------------------------
103 //
104 // Em C é possível «empacotar» código para o poder usar em vários locais. Ao
105 // código assim «empacotado» chama-se um _rotina_. As rotinas em C podem ser
106 // _funções_, se não alterarem o mundo, limitando-se a resultar num valor, ou
107 // _procedimentos_, se agirem sobre algum aspecto do programa ou do mundo que o
108 // rodeia, provocando potencialmente alterações. Neste caso estamos a definir
109 // uma função chamada `factorial()`.
110 //
111 // A interface de uma rotina diz _como a rotina se usa_ e _o que a rotina faz ou
112 // calcula_. Pelo contrário, a implementação de uma rotina especifica _como a
113 // rotina funciona_. A definição de uma rotina, seja ela função ou procedimento,
114 // consiste numa interface, que corresponde ao chamado _cabeçalho_ da rotina, e
115 // numa implementação, que corresponde ao chamado _corpo_ dessa mesma rotina.
116 
117 // ### Cabeçalho da função `factorial()`
118 //
119 // O cabeçalho de uma rotina é parte da sua _interface_. Ou seja, daquilo que é
120 // necessário conhecer para a conseguir utilizar. O cabeçalho de uma rotina
121 // indica:
122 //
123 // - O seu nome &ndash; Neste caso o nome é `factorial`, pois é suposto esta
124 // função calcular e devolver o factorial do valor que lhe for _passado_ como
125 // _argumento_.
126 //
127 // - A lista dos seus _parâmetros_, incluindo o tipo e o nome de cada uma desses
128 // _parâmetros_ &ndash; Os _parâmetros_ são _variáveis locais_ (i.e., visíveis
129 // apenas num contexto local, e.g., no corpo de uma rotina), _automáticas_
130 // (i.e., construídas ao entrar no respectivo contexto e destruídas ao dele
131 // sair), que têm a particularidade de ser inicializadas com o valor do
132 // _argumento_ correspondente sempre que a rotina é _executada_ como resultado
133 // de uma sua _invocação_.
134 //
135 // - O tipo do valor _devolvido_ quando a rotina _retorna_ &ndash; As funções
136 // (bem como alguns procedimentos) são invocadas para se obter um dado valor
137 // como resultado. Esse valor é calculado pela rotina e _devolvido_ quando a
138 // rotina terminar, i.e., quando se _retorna_ ao local onde essa rotina foi
139 // invocada.
140 //
141 // A função `factorial()` recebe um `int` como argumento. Esse `int` é guardado
142 // dentro do parâmetro `n`, que, aliás, é uma _constante_, pois o _qualificador_
143 // `const` altera a natureza de `n`, que deixa de ser uma variável e se torna
144 // constante. É muito importante perceber a diferença entre um _argumento_, que
145 // é uma expressão ou valor passado quando se _invoca_ uma rotina, e um
146 // _parâmetro_, que é uma variável ou constante especial que, no início da
147 // execução da rotina, é inicializada automaticamente com o valor do argumento
148 // passado aquando da invocação da mesma rotina. Mais abaixo se verão invocações
149 // desta função.
150 //
151 // Quando a função `factorial()` termina, altura em que o fluxo de execução
152 // _retorna_ ao ponto de invocação, _devolve_ um valor. Esse valor é o factorial
153 // do valor recebido como argumento, e é do tipo `int`, neste caso. Isso é
154 // indicado pela presença da _palavra-chave_ `int` antes do nome da função _no
155 // cabeçalho da definição da função_.
156 int factorial(const int n)
157 // ### Corpo da função `factorial()`
158 //
159 // O corpo de uma rotina contém a sua implementação. Em C, o corpo de uma rotina
160 // consiste sempre num _bloco de instruções_, ou seja, numa sequência de
161 // instruções envoltas por chavetas.
162 //
163 // A implementação que usamos para a função `factorial()` nasce numa definição
164 // recursiva da noção de factorial.
165 //
166 // O factorial de um número ![n](http://bit.ly/Z4ELdk) representa-se por
167 // ![n](http://bit.ly/Z4ELdk) e corresponde ao produto de todos os números
168 // inteiros entre 0 e ![n](http://bit.ly/Z4ELdk), se
169 // ![n>0](http://bit.ly/ZteSVa), tendo, por definição, o valor 1 quando
170 // ![n=0](http://bit.ly/10afb7o). Ou seja, se ![n>0](http://bit.ly/ZteSVa),
171 // ![n!=n(n-1)...1](http://bit.ly/12553hX). É fácil concluir daqui que
172 // ![n!=n(n-1)!](http://bit.ly/113hsrr), desde que ![n>0](http://bit.ly/ZteSVa).
173 // Ou seja,
174 //
175 // ![Definição recursiva do factorial](http://bit.ly/18B6xTd)
176 //
177 // A implementação é consequência directa desta definição.
178 //
179 // Note que esta função não faz qualquer esforço para lidar com valores
180 // inválidos do seu argumento. Se se passar à função um valor negativo ou um
181 // valor cujo factorial seja demasiado grande para representar num `int`, a
182 // função pura e simplesmente não funciona. Isto é um erro de programação,
183 // naturalmente. Uma forma de resolver este erro seria:
184 //
185 // 1. Incluir no topo deste ficheiro, para além dos ficheiros de cabeçalho já
186 // incluídos, o ficheiro de cabeçalho `assert.h`.
187 //
188 // 2. Alterar o corpo da função se modo a fazer a verificação da validade do seu
189 // argumento usando uma rotina especial (na realidade uma _macro_ do
190 // pré-processador) chamada `assert()`:
191 //
192 // ```C
193 // {
194 // assert(0 <= n && n <= 12);
195 //
196 // if (n == 0)
197 // return 1;
198 // return n * factorial(n - 1);
199 // }
200 // ```
201 //
202 // Note que o limite superior para o valor do argumento é muito baixo: 12. Isso
203 // deve-se a dois factos:
204 //
205 // 1. Os `int` em C, nos nossos ambientes típicos, são guardados em apenas 32
206 // _bit_ interpretados em complemento para dois, pelo que um inteiro só pode
207 // tomar valores entre ![-2^31](http://bit.ly/16CVI6B) e
208 // ![2^31-1](http://bit.ly/12AavZ0), ou seja, entre
209 // -2&thinsp;147&thinsp;483&thinsp;648 e 2&thinsp;147&thinsp;483&thinsp;647.
210 //
211 // 2. O factorial é uma função de crescimento muito rápido, pelo que 13!, com o
212 // valor 6&thinsp;227&thinsp;020&thinsp;800, excede já o limite superior dos
213 // `int`.
214 
215 //
216 {
217 // Começamos por verificar se se aplica o caso especial em que o valor de `n` é
218 // zero.
219  if (n == 0)
220 // Se se aplicar, retorna-se imediatamente (i.e., termina-se a função,
221 // retornando ao ponto de invocação) _devolvendo_ o valor 1.
222  return 1;
223 // No caso contrário, retorna-se devolvendo o produto entre o valor de `n` e o
224 // valor devolvido por... uma nova invocação da mesma função, embora com um
225 // valor mais pequeno passado como argumento, ou seja, _n - 1_ em vez de _n_. A
226 // uma invocação deste tipo chama-se uma _invocação recursiva_. Note-se que o
227 // funcionamento desta função decorre do facto de, cada vez que a função é
228 // invocada, as respectivas variáveis locais serem construídas na _pilha_ (uma
229 // zona especial da memória de um _processo_ em execução). Isso significa que,
230 // durante a execução da função decorrente de uma invocação `factorial(4)`, por
231 // exemplo, chega a haver cinco instâncias ou versões do parâmetro _n_ em
232 // existência _em simultâneo_.
233  return n * factorial(n - 1);
234 }

236 // A rotina principal `main()`
237 // ---------------------------
238 //
239 // Todos os programas escritos em C têm de definir um procedimento `main()`,
240 // i.e., com o nome `main`. A execução de um programa escrito em C começa sempre
241 // com a invocação dessa rotina, que por isso é o ponto de entrada no programa.
242 //
243 // O nosso programa tem como objectivo calcular o factorial de um valor
244 // introduzido pelo utilizador no teclado. Um possível exemplo de interacção do
245 // programa com o utilizador é o seguinte:
246 //
247 // ```
248 // mises:hello_world mmsequeira$ bin/Debug/hello_world
249 // Hello world! Let's calculate the factorial of a number.
250 // Please enter a non-negative integer: 10
251 // 10! = 3628800
252 // ```
253 //
254 // Tal como referido anteriormente, este programa está errado, pois não faz
255 // qualquer tentativa de lidar com erros do utilizador. Por exemplo, se o
256 // utilizador não introduzir um número, mas sim o seu nome, o resultado é
257 // inesperado:
258 //
259 // ```
260 // mises:hello_world mmsequeira$ bin/Debug/hello_world
261 // Hello world! Let's calculate the factorial of a number.
262 // Please enter a non-negative integer: Manuel
263 // 32767! = 0
264 // ```
265 //
266 // Note que o valor 32767 nada tem de especial. Quando uma leitura falha, é
267 // usual que a variável na qual se guardaria o valor lido seja deixada sem
268 // qualquer alteração. Não tendo sido inicializada, o valor que contém é
269 // indefinido, no sentido em que pode ser qualquer valor inteiro. O facto de o
270 // valor ter sido 32767 reflecte apenas o ambiente particular de compilação e
271 // execução do programa usado ao gerar esta documentação.
272 //
273 // Também ocorre um erro, embora de outro tipo, se o utilizador introduzir um
274 // valor negativo:
275 //
276 // ```
277 // mises:hello_world mmsequeira$ bin/Debug/hello_world
278 // Hello world! Let's calculate the factorial of a number.
279 // Please enter a non-negative integer: -1
280 // Segmentation fault: 11
281 // ```
282 //
283 // Neste caso o erro ocorre porque, partindo de um valor negativo, só se
284 // atingiria o caso especial da definição recursiva, i.e., o caso em que o
285 // argumento da função é 0, depois de passar por todos os possíveis valores
286 // negativos e positivos dos inteiros (quando se subtrai um ao mais pequeno dos
287 // inteiros obtém-se o maior dos inteiros, por estranho que pareça). Trata-se de
288 // um número demasiado grande de execuções recursivas todas _em curso_, pelo que
289 // o espaço de memória, chamado «pilha» (_stack_), reservado pelo sistema
290 // operativo para guardar informação sobre as rotinas em execução, incluindo as
291 // respectivas variáveis locais, acaba por se esgotar, levando o programa a
292 // _abortar_.
293 //
294 // Finalmente, outro erro deste programa é não fazer qualquer esforço para
295 // verificar se o valor máximo dos valores do tipo _int_ é ultrapassado durante
296 // o cálculo do factorial. Isso leva, mais uma vez, a resultados inesperados,
297 // mesmo perante respostas aparentemente válidas do utilizador:
298 //
299 // ```
300 // mises:hello_world mmsequeira$ bin/Debug/hello_world
301 // Hello world! Let's calculate the factorial of a number.
302 // Please enter a non-negative integer: 17
303 // 17! = -288522240
304 // ```
305 //
306 // Em todos estes casos, mesmo nos dois primeiros, trata-se de erros de
307 // programação: um programa _tem_ de saber lidar com os seus próprios limites e
308 // com quaisquer erros do utilizador. Se não o fizer, _está errado_. Ou seja,
309 // este programa _está errado_, pois optámos por não fazer qualquer verificação
310 // de erros, de modo a mantê-lo simples.
311 //
312 // Uma versão correcta do código é apresentada mais abaixo, com alguns
313 // comentários.
314 
315 // Neste caso declaramos a rotina como não possuindo qualquer parâmetro: daí a
316 // palavra-chave `void` que colocámos entre parênteses após o nome da rotina.
317 // (Note- se, o entanto, que poderíamos ter declarado a rotina de forma
318 // diferente, de modo a poder receber argumentos vindos da linha de comandos.)
319 //
320 // Declaramos também a rotina como devolvendo um valor do tipo `int`. Este tipo
321 // de devolução é obrigatório, no caso da rotina `main()`. O valor devolvido
322 // pela rotina quando retorna é usado pela entidade que executou o programa para
323 // saber se a execução teve ou não sucesso.
324 int main(void)
325 {
326  // Começamos por escrever no ecrã o nome e o propósito do programa. Para
327  // isso, invocamos o procedimento `printf()` passando-lhe como argumento
328  // uma _cadeia de caracteres_, neste caso uma _cadeia de caracteres
329  // literal_, pois usamos a notação do C das cadeias de caracteres
330  // literais. Ou seja, uma sequência de caracteres escritos entre aspas.
331  // O único caso especial aqui é o da sequência de caracteres `\n`.
332  // Quando o C encontra o caractere `\` numa cadeia de caracteres, não o
333  // interpreta literalmente: escapa para um modo de interpretação
334  // diferente, em que os caracteres que se sequem ao `\` têm um
335  // significado especial. Neste caso usamos a sequência `\n`, que
336  // significa o caractere _fim-de-linha_. Este caractere termina a linha
337  // corrente e dá início a uma nova linha de caracteres no ecrã.
338  printf("Hello world! Let's calculate the factorial of a number.\n");
339 
340  // Uma vez que o objectivo do programa é calcular o factorial de um
341  // número, começamos por pedir ao utilizador para introduzir um número
342  // inteiro não negativo. Note que neste caso não terminamos a linha,
343  // pois pretendemos que a resposta do utilizador surja visualmente à
344  // frente do texto que escrevemos. Em vez disso, escrevemos um espaço,
345  // para que o número introduzido pelo utilizador não surja «colado» ao
346  // nosso texto. Note ainda que a leitura dos caracteres introduzidos
347  // pelo utilizador só é realizada depois de este carregar em «enter», o
348  // que terá o efeito visual de terminar esta linha.
349  printf("Please enter a non-negative integer: ");
350 
351  // Antes de proceder à leitura propriamente dita, temos de definir uma
352  // variável para guardar o valor introduzido pelo utilizador. Chamamos a
353  // essa variável `number` e atribuímos-lhe o tipo _int_, compatível com
354  // o parâmetro de entrada da função _factorial()_ já definida. A
355  // definição de uma variável em C começa sempre por um tipo. Nos casos
356  // mais simples, esse tipo é seguido do nome da variável em definição,
357  // tal como fazemos aqui. Este formato de definição é exactamente o
358  // mesmo que usámos no cabeçalho da função `factorial()`, aliás.
359  int number;
360 
361  // Para proceder à leitura do número introduzido pelo utilizador
362  // invocamos o procedimento `scanf()`, passando-lhe dois argumentos. O
363  // primeiro é uma cadeia de caracteres literal a que se chama _formato_
364  // e que contém uma chamada _especificação de conversão_, iniciada pelo
365  // caractere `&`. A leitura é controlada por esta cadeia. Quando o
366  // procedimento encontra a especificação de conversão `%d`, procede à
367  // leitura dos próximos caracteres da entrada admitindo que estes
368  // correspondem à representação decimal usual de um número inteiro.
369  scanf("%d", &number);
370 
371  // Finalmente, podemos calcular o factorial pretendido e escrever o
372  // resultado no ecrã. Para isso invocamos o procedimento `printf()`
373  // passando-lhe três argumentos.
374  //
375  // O primeiro argumento, `"%d! = %d\n"`, é uma cadeia de caracteres de
376  // formação. Esta cadeia da caracteres é interpretada pelo procedimento
377  // `printf()`, que imprime directamente no ecrã os caracteres da cadeia
378  // de formação, excepto quando encontra _especificações de formatação_
379  // que, tal como no caso do procedimento `scanf()`, são introduzidas
380  // pelo caractere `%`. Neste caso temos duas especificações de
381  // formatação, ambas `%d`.
382  //
383  // Quando o procedimento `printf()` encontra a primeira especificação de
384  // formatação, transforma o valor do segundo argumento, que tem de ser
385  // um `int`, uma vez que após o caractere `%` se encontra o caractere
386  // `d`, numa sequência de caracteres e escreve-os no ecrã. Isso leva a
387  // que o valor da variável `number` seja escrito no ecrã, na notação
388  // decimal usual. Quando o procedimento `printf()` encontra a segunda
389  // especificação de formatação, transforma o valor do terceiro argumento
390  // numa sequência de caracteres e escreve-os no ecrã. Isso leva a que o
391  // valor devolvido pela função `factorial()` quando invocada com o
392  // argumento `number` seja escrito no ecrã, na notação decimal usual.
393  //
394  // Por exemplo, admitindo que o valor de `number` é 5, então esta
395  // instrução escreve no ecrã a seguinte linha:
396  //
397  // ```
398  // 5! = 120
399  // ```
400  printf("%d! = %d\n", number, factorial(number));
401 
402  // Terminada a impressão do factorial calculado, podemos terminar o
403  // programa retornando da execução da rotina `main()`. Uma vez que a
404  // execução teve sucesso (ou pelo menos admite-se que sim...),
405  // devolvemos o valor especial `EXIT_SUCCESS` ao retornar.

412 //
413 // ```C
414 // #include <stdio.h>

428 // int main(void)

447 //
448 // printf("%d! = %d\n", number, factorial(number));
449 //
450 // return EXIT_SUCCESS;
451 // }
452 // ```
453 //
454 // Alguns comentários:
455 //
456 // 1. É um bom exercício sobre a linguagem C tentar compreender este código do
457 // princípio ao fim.
458 //
459 // 2. A inclusão de `stdbool.h` deve-se à necessidade de usar o valor booleano
460 // `true` como _guarda_ do ciclo `while`.
461 //
462 // 3. A leitura do valor do teclado é feita dentro de um ciclo que permite ao
463 // utilizador corrigir os seus próprios erros.
464 //
465 // 4. O ciclo usado é um _ciclo infinito_ que, ao contrário do que o nome
466 // sugere, termina logo que o utilizador introduza um valor válido. A utilização
467 // de um ciclo infinito `while(true)` deve-se ao facto de as condições de
468 // paragem do ciclo se encontrarem a meio do seu _passo_.
469 //
470 // 5. O ciclo tem duas condições de paragem. A primeira ocorre quando a leitura
471 // tem sucesso e o valor lido é válido. A função `scanf()` devolve o número de
472 // atribuições a itens de entrada realizadas com sucesso. No nosso caso podemos
473 // saber se foi lido um valor inteiro com sucesso verificando se o valor
474 // devolvido foi 1. Para saber se o valor lido é válido, verificamos se se
475 // encontra entre 0 e 12. Caso tudo tenha corrido bem, o ciclo é interrompido
476 // através da instrução `break`.
477 //
478 // 6. A segunda condição de paragem ocorre quando, depois de termos tentado ler
479 // e de não termos tido sucesso, verificamos que o canal de leitura `stdin`
480 // ficou marcado como tendo sido atingido o fim do ficheiro (a entrada de um
481 // programa pode ser redireccionada de modo a ser feita a partir de um ficheiro)
482 // ou tendo ocorrido um outro erro de leitura. Nesse caso terminamos o programa
483 // devolvendo o valor especial `EXIT_FAILURE` para comunicar este erro à
484 // entidade que executou o programa, isto depois de escrevermos uma mensagem de
485 // erro apropriada no canal de escrita de erros `stderr`.
486 //
487 // 7. Quando detectamos um erro de leitura não fatal, avisamos o utilizador do
488 // problema e, depois de descartar a linha errónea introduzida pelo utilizador,
489 // repetimos o passo do ciclo.
490 //
491 // 8. Para descartar uma linha completa de caracteres, invocamos o procedimento
492 // `scanf()` com a cadeia de formação `%*[^\n]%*1[\n]`. O significado é o
493 // seguinte:
494 //
495 // - Primeira especificação de conversão, `%*[^\n]`: Ler _e descartar_ (daí o
496 // caractere `*`) todos os caracteres até o próximo fim-de-linha, exclusive.
497 //
498 // - Segunda especificação de conversão, `%*1[\n]`: Ler _e descartar_ (daí o
499 // caractere `*`) exactamente um caractere fim-de-linha. Colocar `\n` na
500 // cadeia de formatação em vez desta especificação de conversão não teria o
501 // resultado pretendido, pois o procedimento `scanf()` interpreta todos os
502 // caracteres «brancos» como uma ordem para ignorar um número arbitrário de
503 // caracteres brancos até ser encontrado o primeiro caractere não branco, o
504 // que levaria a que não surgisse novo pedido de inserção de um número
505 // válido enquanto o utilizador fosse carregando em «enter».
506 //
507 // Se tudo isto lhe pareceu complexo, não se preocupe: é mesmo complexo. :-)
