14 // Coelhos, recursão e iteração
15 // ----------------------------
16 //
17 // Ofereceram ao Sr. Fibonacci um casal de coelhos, recém-nascidos, de um
18 // espécie curiosa: tornam-se férteis ao fim de um mês de vida, concebem logo
19 // que possível, têm uma gestação de exactamente um mês, cada ninhada consiste
20 // sempre num casalinho de coelhos. Nunca morrem. O Sr. Fibonacci preocupou-se:
21 // teria comida que chegue para os coelhos? Pôs-se a calcular a evolução do
22 // número de casais de coelhos ao longo do tempo:
23 //
24 // 0. No mês zero o Sr. Fibonacci vivia feliz, sem qualquer coelho. Mal
25 // calculava o seu destino...
26 //
27 // 1. No primeiro mês há um casal de coelhos inférteis.
28 //
29 // 2. No segundo mês há um casal de coelhos, já férteis e, claro está, com a
30 // coelha no início da gestação de mais um casalinho.
31 //
32 // 3. No terceiro mês há o casal de coelhos original, fértil, e já no início da
33 // gestação de mais um casalinho de coelhos. E há, claro está, um casal recém-
34 // nascido e, por isso, infértil. Ou seja, há dois casais de coelhos, um fértil
35 // e outro infértil.
36 //
37 // 4. No quarto mês há os dois casais do mês anterior, já férteis, adicionados
38 // de um casal recém-nascido, resultado do acasalamento do único casal fértil do
39 // mês anterior. Ou seja, há três casais de coelhos, dois férteis e um infértil.
40 //
41 // 5. No quinto mês há os três casais do mês anterior, já férteis, adicionados
42 // de dois casais recém-nascidos, resultado do acasalamento entre os dois casais
43 // férteis do mês anterior. Ou seja, há cinco casais de coelhos, três férteis e
44 // dois inférteis.
45 //
46 // O Sr. Fibonacci concluiu rapidamente que, a partir do terceiro mês,
47 //
48 // - o número de casais férteis num mês é sempre igual ao número total de casais
49 // no mês anterior e
50 //
51 // - o número total de casais num mês é sempre igual à soma do número total de
52 // casais existente no mês anterior com o número de casais recém-nascidos, que
53 // é igual ao número de casais férteis no mês anterior e que, por isso, é
54 // igual ao número total de casais existentes dois meses antes.
55 //
56 // Sendo ![F(n)](http://bit.ly/ZCdcqY) o número de casais de coelhos no mês
57 // ![n](http://bit.ly/Z4ELdk), então, desde que ![n>2](http://bit.ly/10vWeOB),
58 // ![F(n)=F(n-2)+F(n-1)](http://bit.ly/10zLPR3). Quando
59 // ![n=1](http://bit.ly/14HsgeM) ou ![n=2](http://bit.ly/YVICg2),
60 // ![F(n)=1](http://bit.ly/13V3GYG). Ou seja, esta sucessão pode ser descrita de
61 // forma _recursiva_, ou seja, definindo os termos da sucessão em função de
62 // outros termos da mesma sucessão.
63 //
64 // O que pretendemos aqui é implementar uma função que receba o mês como
65 // argumento e devolva o número de casais existentes nesse mês. Como não
66 // fornecemos apenas uma implementação, mas várias, com diferentes eficiências,
67 // os nomes dessas funções não poderão ser simplesmente `fibonacci()`, incluindo
68 // no seu nome um sufixo que identifica o tipo de implementação usado. Pode
69 // argumentar-se que, sendo a eficiência parte da interface de um módulo, então
70 // é razoável usar esses sufixos. No entanto, seria preferível, se o nosso
71 // objectivo não fosse o estudo dos algoritmos em si, mas a criação de
72 // ferramentas para utilização futura ou para disponibilização a terceiros, usar
73 // sufixos que fossem reveladores da eficiência da função, e não da sua
74 // implementação.
75 //
76 // Antes, porém, de passar às implementações, é preciso dizer que o presente
77 // oferecido ao Sr. Fibonacci está realmente envenenado. Com efeito, a sucessão
78 // de Fibonacci cresce de forma extremamente rápida. Assim, no mês 120, ou seja,
79 // dez anos depois da oferta do casal original, o número de casais a alimentar é
80 // exactamente 5&thinsp;358&thinsp;359&thinsp;254&thinsp;990&thinsp;966&thinsp;640&thinsp;871&thinsp;840.
81 // Este valor é demasiado grande para os `int` nas nossas máquinas habituais,
82 // que têm 32 _bits_, ou mesmo para os _long_, com 64 _bits_ nessas mesmas
83 // máquinas. Por outro lado, a precisão limitada dos tipos de virgula flutuante
84 // pode pôr problemas complicados ao cálculo dos termos da sucessão (já
85 // tínhamos dito que dos tipos de vírgula flutuante se foge como diabo da
86 // cruz?). O código abaixo recorre ao tipo `long`, numa tentativa de evitar que
87 // se atinjam muito cedo os limites dos inteiros. É uma má solução, no entanto.
88 // Uma boa solução passaria pela utilização de um tipo capaz de representar
89 // números inteiros de dimensão arbitrária (desde que haja memória para o
90 // guardar, bem entendido). A implementação de um tal tipo é um excelente
91 // exercício de algoritmos e estruturas de dados, mas não o resolveremos aqui.

93 // Implementação
94 // -------------
95 //
96 // Como se referiu, usamos `long` como tipo de devolução das funções de cálculo
97 // dos termos da sucessão de Fibonacci. Será que isso adianta muito, em relação
98 // à utilização de _int_? Fazendo algumas experiências, é fácil verificar que,
99 // em máquinas em que os `int` têm 32 _bits_, ou seja, em máquinas em que o
100 // maior valor representável num `int` é 2&thinsp;147&thinsp;483&thinsp;647, o
101 // maior termo da sucessão de Fibonacci representável num `int` é o termo 46, ou
102 // seja, ![F(46)](http://bit.ly/XgffV7), cujo valor é
103 // 1&thinsp;836&thinsp;311&thinsp;903. Isto é, usando `int` só conseguimos obter
104 // valores correctos para os primeiros 47 termos da sucessão. Usando `long`, é
105 // fácil verificar que, em máquinas em que os `long` têm 64 _bits_, ou seja, em
106 // máquinas em que o maior valor representável num `long` é
107 // 9&thinsp;223&thinsp;372&thinsp;036&thinsp;854&thinsp;775&thinsp;807, o maior
108 // termo da sucessão de Fibonacci representável num `long` é o termo 92, ou
109 // seja, ![F(92)](http://bit.ly/12kthcr), cujo valor é
110 // 7&thinsp;540&thinsp;113&thinsp;804&thinsp;746&thinsp;346&thinsp;429. Isto é,
111 // usando `long` só conseguimos obter valores correctos para os primeiros 93
112 // termos da sucessão. Melhor que os 47 se se usasse `int`, mas ainda assim um
113 // valor bem pequeno.
114 //
115 // O maior termo calculável usando `long`, 92, deve ser usado nas pré-condições
116 // das funções abaixo. Usar o valor literal nesse código é muito má ideia, pois
117 // torna mais difícil a compreensão do código. Afinal, haverá que perceba
118 // imediatamente o significado desse valor? Para resolver o problema definimos
119 // uma constante para guardar esse valor. Uma vez que esta constante será usada
120 // para dimensionar _arrays_ que pretendemos inicializar usando as usuais listas
121 // de inicializadores do C, não podemos usar a forma mais óbvia de definir
122 // constantes em C, i.e., não podemos usar
123 //
124 // ```C
125 // const int maximum_term_fitting_a_long = 92;
126 // ```
127 //
128 // tendo, pelo contrário de usar uma macro do pré-processador.
129 #define MAXIMUM_TERM_FITTING_A_LONG 92

180 // #### Definição
181 //
182 // Esta implementação segue a estratégia mais comum das soluções recursivas:
183 // decompor o problema em problemas mais simples, resolver esses problemas mais
184 // simples usando exactamente a mesma estratégia, recursivamente, e compor as
185 // soluções dos subproblemas de modo a obter a solução do problema como um todo.
186 // Uma solução recursiva, como esta, precisa de possuir casos de paragem (ou
187 // casos especiais), para os quais a solução é imediata, não sendo necessária
188 // qualquer decomposição adicional. A definição recursiva da sucessão de
189 // Fibonacci aparenta adequar-se bem a esta estratégia:
190 //
191 // ![F(n)={0, se n = 0, 1, se n = 1, e F(n-2) + F(n-1), se n >
192 // 1}](http://bit.ly/13TPIpP)
193 long stupidly_recursive_fibonacci(int n)
194 {
195  // Em primeiro lugar, verificamos as pré-condições e lidamos com as
196  // violações que forem detectadas.
197  assert(n >= 0);
198  assert(n <= MAXIMUM_TERM_FITTING_A_LONG);
199 
200  // Depois, lidamos com os dois casos especiais da sucessão. Estes casos
201  // servem também para garantir que as invocações recursivas desta função
202  // acabarão sempre por terminar (em tempo finito).
203  if (n == 0)
204  return 0L;
205  if (n == 1)
206  return 1L;
207 
208  // O caso geral é simplesmente a devolução da soma dos termos anterior
209  // da sucessão. Para os obter, invocamos a própria função com os dois
210  // valores inteiros anteriores a `n`. O problema desta implenentação é
211  // que se está a reduzir o problema de calcular
212  // ![F(n)](http://bit.ly/ZCdcqY) aos subproblemas de calcular
213  // ![F(n-2)](http://bit.ly/14HrTAK) e de calcular
214  // ![F(n-1)](http://bit.ly/13V2Vid), _mas estes subproblemas não são
215  // independentes!_
216  return stupidly_recursive_fibonacci(n - 2)
217  + stupidly_recursive_fibonacci(n - 1);
218 }

243 long recursive_fibonacci(int n)
244 {
245  assert(n >= 0);
246  assert(n <= MAXIMUM_TERM_FITTING_A_LONG);
247 
248  // Esta implementação evita os cálculos repetidos dos mesmos termos da
249  // sucessão de Fibonacci usando uma memória auxiliar. A ideia é calcular
250  // cada termo da sucessão uma única vez, guardar o seu valor em memória
251  // e, das próximas vezes que o termo for solicitado, usar o valor
252  // guardado em memória. Isto implica, naturalmente, que a memória usada
253  // persista para além do âmbito restrito de uma execução da função. Ou
254  // seja, não podemos de forma nenhuma usar variáveis locais
255  // _automáticas_, pois estas são construídas quando a instrução que as
256  // define é executada, e destruídas quando se atinge o final do bloco
257  // onde a sua definição se encontra. Em vez de partir daqui as variáveis
258  // globais, cujo tempo de vida abarca todo o tempo de execução do
259  // programa, e que são visíveis em todo o programa, preferimos usar
260  // variáveis locais _estáticas_. Estas variáveis definem-se usando o
261  // _qualificador_ `static`, duram desde a execução da instrução que as
262  // define até o final do programa e têm a mesma visibilidade restrita
263  // que qualquer variável local tem. Note-se que a instrução de definição
264  // de qualquer variável local estática é executada uma única vez,
265  // nomeadamente a primeira vez que o fluxo de execução do programa passa
266  // por essa instrução. A partir daí a instrução de definição, com a
267  // respectiva inicialização, é ignorada. Logo, a inicialização de uma
268  // variável local estática _é feita uma única vez_ durante a execução do
269  // programa.
270  //
271  // Uma vez que o número de termos da sucessão de Fibonacci calculáveis
272  // através desta função é limitado, dado que ela devolve valores `long`,
273  // só se podendo calcular o valor dos termos entre 0 e
274  // `MAXIMUM_TERM_FITTING_A_LONG`, podemos reservar espaço para todos os
275  // termos num _array_ `F` de `long`. A utilização de uma macro
276  // `MAXIMUM_TERM_FITTING_A_LONG` na expressão para cálculo do
277  // comprimento do _array_ permite-nos usar inicializadores do C. Dessa
278  // forma, podemos inicializar a memória dos termos da sucessão de
279  // Fibonacci com os dois primeiros termos dessa sucessão, que têm os
280  // valores 0 e 1, respectivamente, correspondentes aos termos com
281  // índices 0 e 1.
282  static long F[MAXIMUM_TERM_FITTING_A_LONG + 1] = {0, 1};
283  // Não basta definir o espaço de memória dos termos da sucessão: é
284  // necessário definir uma variável que guarde em cada instante quantos
285  // termos estão já calculados. Neste caso inicializa-se a variável com o
286  // valor 2, uma vez que se inicializou a memória com os dois primeiros
287  // termos da sucessão de Fibonacci.
288  static int number_of_calculated_terms = 2;
289 
290  // Estamos agora preparados para indagar o valor do termo da sucessão de
291  // Fibonacci solicitado. Se o termo já constar na memória de termos
292  // calculados, i.e., se o valor `n` for inferior a
293  // `number_of_calculated_terms` limitamo-nos a devolver o valor
294  // memorizado.
295  if (n < number_of_calculated_terms)
296  return F[n];
297 
298  // O caso geral passa por
299  //
300  // 1. invocar recursivamente a própria função para obter os termos `n -
301  // 2` e `n - 1`,
302  //
303  // 2. calcular o termo `n` como soma dos dois termos anteriores
304  // calculados,
305  //
306  // 3. guardar esse novo termo na memória, de modo a não precisar de ser
307  // recalculado, e
308  //
309  // 4. terminar a função, devolvendo o valor do novo termo ao retornar ao
310  // ponto de invocação da função.
311  //
312  // Estes passos estão condensados em apenas duas linhas de código. Essa
313  // condensação seria considerada uma má prática em quase todas as
314  // linguagens de programação, mas este tipo de código é idiomático em C,
315  // pelo que preferimos apresentá-lo desta forma. Seja como for, uma
316  // forma ortodoxa de escrever este código seria:
317  //
318  // ```C
319  // F[n] = recursive_fibonacci(n - 2) + recursive_fibonacci(n - 1);
320  // number_of_calculated_terms++;
321  //
322  // return F[n];
323  // ```
324  //
325  // Alternativamente, podemos considerar que o caso especial é aquele em
326  // que o termo pretendido _não_ foi ainda calculado, o que nos levaria
327  // ao seguinte código, bem mais legível, que substitui não apenas as
328  // duas últimas linhas do código apresentado, mas também as duas
329  // anteriores:
330  //
331  // ```C
332  // if (number_of_calculated_terms <= n) {
333  // F[n] = recursive_fibonacci(n - 2) +
334  // recursive_fibonacci(n - 1);
335  // number_of_calculated_terms++;
336  // }
337  //
338  // return F[n];
339  // ```
340  number_of_calculated_terms++;
341  return F[n] = recursive_fibonacci(n - 2) + recursive_fibonacci(n - 1);
342 }

425 long iterative_fibonacci(int n)
426 {
427  assert(n >= 0);
428  assert(n <= MAXIMUM_TERM_FITTING_A_LONG);
429 
430  if (n == 0)
431  return 0L;
432  if (n == 1)
433  return 1L;
434 
435  long previous_term = 1L;
436  long current_term = 1L;
437  for(int i = 2; i != n; i++) {
438  current_term += previous_term;
439  previous_term = current_term - previous_term;
440  }
441 
442  return current_term;
443 }
