15 // Coelhos, recursão e iteração
16 // ----------------------------
17 //
18 // Ofereceram ao Sr. Fibonacci um casal de coelhos, recém-nascidos, de um
19 // espécie curiosa: tornam-se férteis ao fim de um mês de vida, concebem logo
20 // que possível, têm uma gestação de exactamente um mês, cada ninhada consiste
21 // sempre num casalinho de coelhos. Nunca morrem. O Sr. Fibonacci preocupou-se:
22 // teria comida que chegue para os coelhos? Pôs-se a calcular a evolução do
23 // número de casais de coelhos ao longo do tempo:
24 //
25 // 0. No mês zero o Sr. Fibonacci vivia feliz, sem qualquer coelho. Mal
26 // calculava o seu destino...
27 //
28 // 1. No primeiro mês há um casal de coelhos inférteis.
29 //
30 // 2. No segundo mês há um casal de coelhos, já férteis e, claro está, com a
31 // coelha no início da gestação de mais um casalinho.
32 //
33 // 3. No terceiro mês há o casal de coelhos original, fértil, e já no início da
34 // gestação de mais um casalinho de coelhos. E há, claro está, um casal recém-
35 // nascido e, por isso, infértil. Ou seja, há dois casais de coelhos, um fértil
36 // e outro infértil.
37 //
38 // 4. No quarto mês há os dois casais do mês anterior, já férteis, adicionados
39 // de um casal recém-nascido, resultado do acasalamento do único casal fértil do
40 // mês anterior. Ou seja, há três casais de coelhos, dois férteis e um infértil.
41 //
42 // 5. No quinto mês há os três casais do mês anterior, já férteis, adicionados
43 // de dois casais recém-nascidos, resultado do acasalamento entre os dois casais
44 // férteis do mês anterior. Ou seja, há cinco casais de coelhos, três férteis e
45 // dois inférteis.
46 //
47 // O Sr. Fibonacci concluiu rapidamente que, a partir do terceiro mês,
48 //
49 // - o número de casais férteis num mês é sempre igual ao número total de casais
50 // no mês anterior e
51 //
52 // - o número total de casais num mês é sempre igual à soma do número total de
53 // casais existente no mês anterior com o número de casais recém-nascidos, que
54 // é igual ao número de casais férteis no mês anterior e que, por isso, é
55 // igual ao número total de casais existentes dois meses antes.
56 //
57 // Sendo ![F(n)](http://bit.ly/ZCdcqY) o número de casais de coelhos no mês
58 // ![n](http://bit.ly/Z4ELdk), então, desde que ![n>2](http://bit.ly/10vWeOB),
59 // ![F(n)=F(n-2)+F(n-1)](http://bit.ly/10zLPR3). Quando
60 // ![n=1](http://bit.ly/14HsgeM) ou ![n=2](http://bit.ly/YVICg2),
61 // ![F(n)=1](http://bit.ly/13V3GYG). Ou seja, esta sucessão pode ser descrita de
62 // forma _recursiva_, ou seja, definindo os termos da sucessão em função de
63 // outros termos da mesma sucessão.
64 //
65 // O que pretendemos aqui é implementar uma função que receba o mês como
66 // argumento e devolva o número de casais existentes nesse mês. Como não
67 // fornecemos apenas uma implementação, mas várias, com diferentes eficiências,
68 // os nomes dessas funções não poderão ser simplesmente `fibonacci()`, incluindo
69 // no seu nome um sufixo que identifica o tipo de implementação usado. Pode
70 // argumentar-se que, sendo a eficiência parte da interface de um módulo, então
71 // é razoável usar esses sufixos. No entanto, seria preferível, se o nosso
72 // objectivo não fosse o estudo dos algoritmos em si, mas a criação de
73 // ferramentas para utilização futura ou para disponibilização a terceiros, usar
74 // sufixos que fossem reveladores da eficiência da função, e não da sua
75 // implementação.
76 //
77 // Antes, porém, de passar às implementações, é preciso dizer que o presente
78 // oferecido ao Sr. Fibonacci está realmente envenenado. Com efeito, a sucessão
79 // de Fibonacci cresce de forma extremamente rápida. Assim, no mês 120, ou seja,
80 // dez anos depois da oferta do casal original, o número de casais a alimentar é
81 // exactamente 5&thinsp;358&thinsp;359&thinsp;254&thinsp;990&thinsp;966&thinsp;640&thinsp;871&thinsp;840.
82 // Este valor é demasiado grande para os `int` nas nossas máquinas habituais,
83 // que têm 32 _bits_, ou mesmo para os _long_, com 64 _bits_ nessas mesmas
84 // máquinas. Por outro lado, a precisão limitada dos tipos de virgula flutuante
85 // pode pôr problemas complicados ao cálculo dos termos da sucessão (já
86 // tínhamos dito que dos tipos de vírgula flutuante se foge como diabo da
87 // cruz?). O código abaixo recorre ao tipo `long`, numa tentativa de evitar que
88 // se atinjam muito cedo os limites dos inteiros. É uma má solução, no entanto.
89 // Uma boa solução passaria pela utilização de um tipo capaz de representar
90 // números inteiros de dimensão arbitrária (desde que haja memória para o
91 // guardar, bem entendido). A implementação de um tal tipo é um excelente
92 // exercício de algoritmos e estruturas de dados, mas não o resolveremos aqui.

96 //
97 // Como se referiu, usamos `long` como tipo de devolução das funções de cálculo
98 // dos termos da sucessão de Fibonacci. Será que isso adianta muito, em relação
99 // à utilização de _int_? Fazendo algumas experiências, é fácil verificar que,
100 // em máquinas em que os `int` têm 32 _bits_, ou seja, em máquinas em que o
101 // maior valor representável num `int` é 2&thinsp;147&thinsp;483&thinsp;647, o
102 // maior termo da sucessão de Fibonacci representável num `int` é o termo 46, ou
103 // seja, ![F(46)](http://bit.ly/XgffV7), cujo valor é
104 // 1&thinsp;836&thinsp;311&thinsp;903. Isto é, usando `int` só conseguimos obter
105 // valores correctos para os primeiros 47 termos da sucessão. Usando `long`, é
106 // fácil verificar que, em máquinas em que os `long` têm 64 _bits_, ou seja, em
107 // máquinas em que o maior valor representável num `long` é
108 // 9&thinsp;223&thinsp;372&thinsp;036&thinsp;854&thinsp;775&thinsp;807, o maior
109 // termo da sucessão de Fibonacci representável num `long` é o termo 92, ou
110 // seja, ![F(92)](http://bit.ly/12kthcr), cujo valor é
111 // 7&thinsp;540&thinsp;113&thinsp;804&thinsp;746&thinsp;346&thinsp;429. Isto é,
112 // usando `long` só conseguimos obter valores correctos para os primeiros 93
113 // termos da sucessão. Melhor que os 47 se se usasse `int`, mas ainda assim um
114 // valor bem pequeno.
115 //
116 // O maior termo calculável usando `long`, 92, deve ser usado nas pré-condições
117 // das funções abaixo. Usar o valor literal nesse código é muito má ideia, pois
118 // torna mais difícil a compreensão do código. Afinal, haverá que perceba
119 // imediatamente o significado desse valor? Para resolver o problema definimos
120 // uma constante para guardar esse valor. Uma vez que esta constante será usada
121 // para dimensionar _arrays_ que pretendemos inicializar usando as usuais listas
122 // de inicializadores do C, não podemos usar a forma mais óbvia de definir
123 // constantes em C, i.e., não podemos usar
124 //
125 // ```C
126 // const int maximum_term_fitting_a_long = 92;
127 // ```
128 //
129 // tendo, pelo contrário de usar uma macro do pré-processador.
132 #define MAXIMUM_TERM_FITTING_A_LONG 92

202 // #### Definição
203 //
204 // Esta implementação segue a estratégia mais comum das soluções recursivas:
205 // decompor o problema em problemas mais simples, resolver esses problemas mais
206 // simples usando exactamente a mesma estratégia, recursivamente, e compor as
207 // soluções dos subproblemas de modo a obter a solução do problema como um todo.
208 // Uma solução recursiva, como esta, precisa de possuir casos de paragem (ou
209 // casos especiais), para os quais a solução é imediata, não sendo necessária
210 // qualquer decomposição adicional. A definição recursiva da sucessão de
211 // Fibonacci aparenta adequar-se bem a esta estratégia:
212 //
213 // ![F(n)={0, se n = 0, 1, se n = 1, e F(n-2) + F(n-1), se n >
214 // 1}](http://bit.ly/169sagW)
215 long stupidly_recursive_fibonacci(int n)
216 {
217  // Em primeiro lugar, verificamos as pré-condições e lidamos com as
218  // violações que forem detectadas.
219  assert(n >= 0);
220  assert(n <= MAXIMUM_TERM_FITTING_A_LONG);
221 
222  // Depois, lidamos com os dois casos especiais da sucessão. Estes casos
223  // servem também para garantir que as invocações recursivas desta função
224  // acabarão sempre por terminar (em tempo finito).
225  if (n == 0)
226  return 0L;
227  if (n == 1)
228  return 1L;
229 
230  // O caso geral é simplesmente a devolução da soma dos termos anterior
231  // da sucessão. Para os obter, invocamos a própria função com os dois
232  // valores inteiros anteriores a `n`. O problema desta implenentação é
233  // que se está a reduzir o problema de calcular
234  // ![F(n)](http://bit.ly/ZCdcqY) aos subproblemas de calcular
235  // ![F(n-2)](http://bit.ly/14HrTAK) e de calcular
236  // ![F(n-1)](http://bit.ly/13V2Vid), _mas estes subproblemas não são
237  // independentes!_ Isso faz com que esta implementação seja muitíssimo
238  // ineficiente.
239  return stupidly_recursive_fibonacci(n - 2)
240  + stupidly_recursive_fibonacci(n - 1);
241 }

244 // alguns valores interessantes.
245 //
246 // Em primeiro lugar, uma revelação: a sucessão de Fibonacci pode ser expressa
247 // de forma fechada. «Estamos aqui com tudo isto e a coisa resume-se a uma
248 // fórmula fechada?», podem perguntar. Sim. Mas pode haver boas razões para não
249 // recorrer a essa fórmula: ela obriga-nos a trabalhar com valores de vírgula
250 // flutuante e, por isso, a lidar com as correspondentes limitações de precisão.
251 // As nossas implementações, recorrendo a _long_, são exactas... Podem encontrar
252 // informação sobre a sucessão de Fibonacci, e sobre a sua forma fechada, em
253 // vários locais. Recomendamos os seguintes:
254 //
255 // - [F_n](http://www.wolframalpha.com/input/?i=F_n) no
256 // [Wolfram|Alpha](http://www.wolframalpha.com/) - O Wolfram|Alpha é um motor
257 // de pesquisa com a [Wolfram](http://www.wolfram.com/) por trás (a mesma que
258 // produz o maravilhoso
259 // [Wolfram|Mathematica](http://www.wolfram.com/mathematica/)) só podia ser
260 // excelente. E é-o.
261 //
262 // - [Fibonacci Number](http://mathworld.wolfram.com/FibonacciNumber.html) no
263 // [Wolfram|MathWorld](http://mathworld.wolfram.com/) - O Wolfram|MathWorld é
264 // uma enciclopédia matemática também com a garantia de qualidade da
265 // [Wolfram](http://www.wolfram.com/). Outra excelente fonte de informação.
266 //
267 // - [Fibonacci number](http://en.wikipedia.org/wiki/Fibonacci_number) na
268 // [Wikipédia](http://en.wikipedia.org/).
269 //
270 // A forma fechada da sucessão de Fibonacci é
271 //
272 // ![F_n=(phi^n - psi^n)/sqrt(5)](http://bit.ly/10fxJ6c),
273 //
274 // onde
275 //
276 // ![phi=(1+sqrt(5))/2](http://bit.ly/15hTKsW), que é chamado _número de ouro_,
277 // e
278 //
279 // ![psi=(1-sqrt(5))/2](http://bit.ly/12Elq43).
280 //
281 // Outra forma fechada é
282 //
283 // ![F_n=arred(phi^n/sqrt(5))](http://bit.ly/1254VAx),
284 //
285 // onde
286 //
287 // ![arred(x)](http://bit.ly/10bSbcU)
288 //
289 // é a função que resulta no inteiro mais próximo ou arredondamento de
290 // ![x](http://bit.ly/12ElMrk), sendo que em caso de empate entre dois inteiros
291 // mais próximos se escolhe o inteiro que for par.
292 //
293 // Ambas as formas mostram bem o que já tínhamos observado: o crescimento da
294 // sucessão de Fibonacci é extremamente rápido. Mais precisamente, o crescimento
295 // da sucessão de Fibonacci é _exponencial_, sendo a base da exponencial o
296 // número de ouro e o expoente o número do termo da sucessão.
297 //
298 // Vamos agora calcular o número de invocações da função `recursive_fibonacci()`
299 // que ocorrem durante a cálculo de um dado termo ![n](http://bit.ly/Z4ELdk) da
300 // sucessão de Fibonacci, ou seja, durante a execução da função que acabámos de
301 // definir quando se lhe passa como argumento esse valor
302 // ![n](http://bit.ly/Z4ELdk). Chamemos a esse número de invocações
303 // ![N_n](http://bit.ly/139Mk7p).
304 //
305 // Em primeiro lugar, é evidente que invocar `recursive_fibonacci()` com
306 // ![n](http://bit.ly/Z4ELdk) como argumento leva a uma invocação inicial,
307 // nomeadamente a que acabámos de referir, com ![n](http://bit.ly/Z4ELdk) como
308 // argumento. Se o valor de ![n](http://bit.ly/Z4ELdk) for 0 ou 1, então não é
309 // realizada qualquer outra invocação. Ou seja, ![N_n=1](http://bit.ly/YwMaHD)
310 // quando ![n=0](http://bit.ly/10afb7o) ou ![n=1](http://bit.ly/14HsgeM). Se o
311 // valor de ![n](http://bit.ly/Z4ELdk) for maior que 1, então a função será
312 // invocada recursivamente com os valores ![n-2](http://bit.ly/18mQyLi) e
313 // ![n-1](http://bit.ly/10fBCrR) como argumento, o que resultará num total de
314 // ![1+N_n-2+N_n-1](http://bit.ly/Zxxpzv) invocações. Ou seja, o número de
315 // invocações recursivas da função pode ser expresso também de forma recursiva:
316 //
317 // ![definição recursiva de N_n](http://bit.ly/125adMb)
318 //
319 // Esta definição é muito parecida com a definição recursiva da própria sucessão
320 // de Fibonacci. De facto, é fácil demonstrar que ![N_n =
321 // 2F_n+1-1](http://bit.ly/10zTgWa). Ou seja, o número de invocações cresce
322 // exponencialmente, tal como a própria sucessão de Fibonacci! Não admira que
323 // esta implementação seja absolutamente inaceitável...
324 //
325 // Já agora, podemos também calcular o número de somas realizadas. Seja
326 // ![S_n](http://bit.ly/13g3vTT) o número de somas realizadas quando se invoca
327 // `recursive_fibonacci()` com o argumento ![n](http://bit.ly/Z4ELdk). É fácil
328 // ver que, quando ![n](http://bit.ly/Z4ELdk) é 0 ou 1, a invocação de
329 // `recursive_fibonacci()` resulta em 0 somas. Quando se invoca com
330 // ![n](http://bit.ly/Z4ELdk) maior que 1, realiza-se uma soma dos resultados
331 // das invocações recursivas com os valores ![n-2](http://bit.ly/18mQyLi) e
332 // ![n-1](http://bit.ly/10fBCrR) como argumento, o que resultará num total de
333 // ![1+S_n-2+S_n-1](http://bit.ly/YnkQcQ) somas. Ou seja, o número de somas
334 // realizadas durante a invocação da função pode ser expresso também de forma
335 // recursiva:
336 //
337 // ![definição recursiva de S_n](http://bit.ly/13VO8Ql)
338 //
339 // Esta definição é também muito parecida com a definição recursiva da própria
340 // sucessão de Fibonacci. De facto, é fácil demonstrar que ![S_n =
341 // F_n+1-1](http://bit.ly/139Tdpr). Ou seja, o número de somas cresce
342 // exponencialmente, tal como a própria sucessão de Fibonacci! Isto quando uma
343 // implementação iterativa trivial precisa de 0 somas para calcular o termo 0 da
344 // sucessão e de ![n-1](http://bit.ly/10fBCrR) somas para calcular o termo
345 // ![n](http://bit.ly/Z4ELdk) quando ![n>0](http://bit.ly/ZteSVa). Mais uma vez,
346 // não admira que esta implementação seja absolutamente inaceitável.
347 //
348 // A moral desta estória não é que a recursividade seja naturalmente perversa.
349 // Não o é. As soluções recursivas podem ser tão eficientes quanto as soluções
350 // iterativas. O problema aqui não é a recursividade em si, mas o algoritmo
351 // usado.

391 long recursive_fibonacci(int n)
392 {
393  assert(n >= 0);
394  assert(n <= MAXIMUM_TERM_FITTING_A_LONG);
395 
396  // Esta implementação evita os cálculos repetidos dos mesmos termos da
397  // sucessão de Fibonacci usando uma memória auxiliar. A ideia é calcular
398  // cada termo da sucessão uma única vez, guardar o seu valor em memória
399  // e, das próximas vezes que o termo for solicitado, usar o valor
400  // guardado em memória. Isto implica, naturalmente, que a memória usada
401  // persista para além do âmbito restrito de uma execução da função. Ou
402  // seja, não podemos de forma nenhuma usar variáveis locais
403  // _automáticas_, pois estas são construídas quando a instrução que as
404  // define é executada, e destruídas quando se atinge o final do bloco
405  // onde a sua definição se encontra. Em vez de partir daqui as variáveis
406  // globais, cujo tempo de vida abarca todo o tempo de execução do
407  // programa, e que são visíveis em todo o programa, preferimos usar
408  // variáveis locais _estáticas_. Estas variáveis definem-se usando o
409  // _qualificador_ `static`, duram desde a execução da instrução que as
410  // define até o final do programa e têm a mesma visibilidade restrita
411  // que qualquer variável local tem. Note-se que a instrução de definição
412  // de qualquer variável local estática é executada uma única vez,
413  // nomeadamente a primeira vez que o fluxo de execução do programa passa
414  // por essa instrução. A partir daí a instrução de definição, com a
415  // respectiva inicialização, é ignorada. Logo, a inicialização de uma
416  // variável local estática _é feita uma única vez_ durante a execução do
417  // programa.
418  //
419  // Uma vez que o número de termos da sucessão de Fibonacci calculáveis
420  // através desta função é limitado, dado que ela devolve valores `long`,
421  // só se podendo calcular o valor dos termos entre 0 e
422  // `MAXIMUM_TERM_FITTING_A_LONG`, podemos reservar espaço para todos os
423  // termos num _array_ `F` de `long`. A utilização de uma macro
424  // `MAXIMUM_TERM_FITTING_A_LONG` na expressão para cálculo do
425  // comprimento do _array_ permite-nos usar inicializadores do C. Dessa
426  // forma, podemos inicializar a memória dos termos da sucessão de
427  // Fibonacci com os dois primeiros termos dessa sucessão, que têm os
428  // valores 0 e 1, respectivamente, correspondentes aos termos com
429  // índices 0 e 1.
430  static long F[MAXIMUM_TERM_FITTING_A_LONG + 1] = {0, 1};
431  // Não basta definir o espaço de memória dos termos da sucessão: é
432  // necessário definir uma variável que guarde em cada instante quantos
433  // termos estão já calculados. Neste caso inicializa-se a variável com o
434  // valor 2, uma vez que se inicializou a memória com os dois primeiros
435  // termos da sucessão de Fibonacci.
436  static int number_of_calculated_terms = 2;
437 
438  // Estamos agora preparados para indagar o valor do termo da sucessão de
439  // Fibonacci solicitado. Se o termo já constar na memória de termos
440  // calculados, i.e., se o valor `n` for inferior a
441  // `number_of_calculated_terms` limitamo-nos a devolver o valor
442  // memorizado.
443  if (n < number_of_calculated_terms)
444  return F[n];
445 
446  // O caso geral passa por
447  //
448  // 1. invocar recursivamente a própria função para obter os termos `n -
449  // 2` e `n - 1`,
450  //
451  // 2. calcular o termo `n` como soma dos dois termos anteriores
452  // calculados,
453  //
454  // 3. guardar esse novo termo na memória, de modo a não precisar de ser
455  // recalculado, e
456  //
457  // 4. terminar a função, devolvendo o valor do novo termo ao retornar ao
458  // ponto de invocação da função.
459  //
460  // Estes passos estão condensados em apenas duas linhas de código. Essa
461  // condensação seria considerada uma má prática em quase todas as
462  // linguagens de programação, mas este tipo de código é idiomático em C,
463  // pelo que preferimos apresentá-lo desta forma. Seja como for, uma
464  // forma ortodoxa de escrever este código seria:
465  //
466  // ```C
467  // F[n] = recursive_fibonacci(n - 2) + recursive_fibonacci(n - 1);
468  // number_of_calculated_terms++;
469  //
470  // return F[n];
471  // ```
472  //
473  // Alternativamente, podemos considerar que o caso especial é aquele em
474  // que o termo pretendido _não_ foi ainda calculado, o que nos levaria
475  // ao seguinte código, bem mais legível, que substitui não apenas as
476  // duas últimas linhas do código apresentado, mas também as duas
477  // anteriores:
478  //
479  // ```C
480  // if (number_of_calculated_terms <= n) {
481  // F[n] = recursive_fibonacci(n - 2) +
482  // recursive_fibonacci(n - 1);
483  // number_of_calculated_terms++;
484  // }
485  //
486  // return F[n];
487  // ```
488  number_of_calculated_terms++;
489  return F[n] = recursive_fibonacci(n - 2) + recursive_fibonacci(n - 1);
490 }

494 // #### Documentação
541 // #### Definição
542 long recursive_fibonacci_using_ADT(int n)
543 {
544  assert(n >= 0);
545  assert(n <= MAXIMUM_TERM_FITTING_A_LONG);
546 
547  // Em vez de se um _array_ de `long` como memória para os termos já
548  // calculados da sucessão de Fibonacci, bem como um inteiro indicando a
549  // quantidade de termos memorizados, recorremos aqui ao TAD sucessão de
550  // `long`, representado pela estrutura `struct sequence_of_longs`. No
551  // caso particular da sequência se Fibonacci, a utilização deste TAD não
552  // traz grandes vantagens face à utilização de um _array_, dada a
553  // pequena quantidade de termos da sucessão representáveis no tipo
554  // _long_. No entanto, é um bom exercício recorrer aqui ao TAD,
555  // exercício que nos pode ser útil noutros casos em que o
556  // dimensionamento _a priori_ do _array_ não seja tão fácil.
557  //
558  // Tal como definido o TAD, o código cliente só pode trabalhar com as
559  // sucessões através de ponteiros. Assim, precisamos de definir uma
560  // variável local _estática_ (de modo a que o seu valor persiste entre
561  // invocações da função `recursive_fibonacci_using_ADT`). Dadas as
562  // restrições do C, não podemos usar o construtor `SEQL_new()`
563  // directamente na inicialização, que tem de ser feita usando uma
564  // _expressão constante_. Assim, optámos por inicializar o ponteiro `F`
565  // com o valor especial `NULL`.
566  static struct sequence_of_longs *F = NULL;
567 
568  // Dada a inicialização do ponteiro com `NULL`, podemos agora detectar o
569  // seu valor inicial `NULL` para lhe atribuir o endereço de uma nova
570  // estrutura `struct sequence_of_longs` construída de forma dinâmica
571  // através do construtor `SEQL_new()`. Este código é executado sempre
572  // que a função é invocada, o que é uma infelicidade, mas não há forma
573  // de o evitar, dado que não é possível inicializar o ponteiro `F` para
574  // a sucessão de `long` com o valor devolvido pela função `SEQL_new()`.
575  //
576  // Outro problema associado às limitações do C é o da libertação de
577  // memória. O C não fornece nenhum mecanismo para executar código no
578  // contexto das variáveis locais estáticas no final do programa de modo
579  // a podermos «arrumar a casa», ou seja, libertar recursos que lhes
580  // estejam associados. Neste caso os recursos são apenas duas variáveis
581  // dinâmicas: (a) a `struct sequence_of_long` apontada por `F` e
582  // (embora, como clientes, não o devêssemos precisar de saber) e (b) o
583  // _array_ dinâmico usado internamente pelo TAD para guardar os termos.
584  // Como toda a memória dinâmica associada ao programa em execução é
585  // libertada durante a sua terminação, a nossa violação do princípio de
586  // que quem reserva memória explicitamente a deve também libertar
587  // explicitamente não é dramática.
588  if (F == NULL)
589  F = SEQL_new();
590 
591  // Se o termo já constar na sucessão de `long` com os termos da sucessão
592  // de Fibonacci já calculados, limitamo-nos a devolvê-lo.
593  if (n < SEQL_length(F))
594  return SEQL_term(F, n);
595 
596  // Se o termo não está ainda calculado, há que fazê-lo. Uma vez que,
597  // depois de o calcular, teremos de o adicionar à memória e devolver,
598  // definimos uma variável `F_n` para guardar o valor calculado. Uma vez
599  // que o valor usado para inicializar esta variável depende do valor de
600  // `n`, usamos o operador `?:` do C para discriminar entre as três
601  // diferentes formas de inicialização, evitando ter de recorrer a uma
602  // definição sem inicialização seguida de duas instruções de selecção
603  // encadeadas:
604  //
605  // ```C
606  // long F_n;
607  // if (n == 0)
608  // F_n = 0L;
609  // else if (n == 1)
610  // F_n = 1L;
611  // else
612  // F_n = recursive_fibonacci_using_ADT(n - 2) +
613  // recursive_fibonacci_using_ADT(n - 1);
614  // ```
615  long F_n = n == 0 ? 0L : n == 1 ? 1L :
616  recursive_fibonacci_using_ADT(n - 2) +
617  recursive_fibonacci_using_ADT(n - 1);
618 
619  // Agora que já temos o valor de `F_n` calculado, devemos guardá-lo em
620  // memória, na nossa sucessão de `long`, de modo a não precisar de ser
621  // calculado de novo.
622  SEQL_add(F, F_n);
623 
624  // Finalmente, devolvemos o valor calculado.
625  return F_n;
626 }
